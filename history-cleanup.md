# Git History Cleanup과 민감정보 제거

이 문서는 운영 중 자주 헷갈리는 두 가지를 구분하기 위해 작성했다.

- 지금부터 특정 파일을 추적하지 않게 만드는 것
- 과거 commit 히스토리에서 특정 파일을 아예 지워 버리는 것

둘은 전혀 다른 작업이다. `git rm --cached`는 전자이고, history rewrite는 후자다.

## `git rm --cached`는 정확히 무엇을 하는가

`git rm --cached <path>`는 파일을 작업 디렉터리에서 지우는 명령이 아니라, `index`에서만 제거하는 명령이다.

즉, 의미는 아래와 같다.

- 로컬 파일은 그대로 남긴다
- 다음 commit부터는 Git이 그 파일을 추적하지 않게 만든다
- 이후 push하면 원격 저장소의 최신 상태에서는 그 파일이 삭제된 것처럼 보인다

폴더라면 하위 엔트리까지 index에서 빼야 하므로 `-r`이 필요하다.

```bash
git rm --cached path/to/file
git rm -r --cached path/to/dir
```

보통은 `.gitignore`와 함께 쓴다.

```bash
echo "path/to/file" >> .gitignore
git rm --cached path/to/file
git commit -m "stop tracking local file"
git push
```

## 왜 `--cached`를 붙이는가

`--cached`가 없으면 Git은 추적 대상뿐 아니라 작업 디렉터리의 실제 파일까지 삭제하려고 한다.

반대로 `--cached`를 붙이면:

- Git의 추적 대상에서는 제거된다
- 하지만 내 로컬 파일은 남는다

그래서 로컬 설정 파일, 생성 파일, 민감한 환경 파일을 "형상관리에서는 제외하되 내 컴퓨터에는 남겨야 할 때" 자주 사용한다.

## Push 후 서버에서는 어떻게 보이는가

이 명령을 commit하고 push하면 원격 저장소의 최신 브랜치 상태에서는 해당 파일이 사라진다.

하지만 이것은 어디까지나 "최신 커밋 기준에서 삭제된 것"일 뿐이다. 과거 히스토리까지 사라지는 것은 아니다.

즉, 이전 commit에는 여전히 아래 상태가 유지된다.

- 파일 내용이 남아 있다
- 예전 commit checkout이 가능하다
- 다른 branch나 tag가 그 파일을 계속 가리킬 수 있다
- blob object도 히스토리 안에 존재할 수 있다

따라서 `git rm --cached`는 `추적 중단`이지 `역사 삭제`가 아니다.

## 정말 아예 없애고 싶다면

그 경우에는 현재 브랜치의 부모 쪽으로 돌아가서 다시 push하는 문제가 아니라, 해당 파일이 포함된 commit 히스토리 자체를 rewrite해야 한다.

핵심은 아래다.

- 단순 삭제 commit으로는 부족하다
- `rebase`, `reset`, `commit --amend` 정도로 끝나는 문제가 아닐 수 있다
- 보통은 `git filter-repo` 같은 도구로 전체 히스토리를 재작성한다

예를 들어 특정 파일을 모든 히스토리에서 제거하려면 아래처럼 작업한다.

```bash
git filter-repo --path path/to/file --invert-paths
```

폴더라면:

```bash
git filter-repo --path path/to/dir/ --invert-paths
```

이 작업은 "해당 파일이 없었던 새로운 역사"를 다시 만드는 것에 가깝다.

## 왜 부모 브랜치만 다시 밀어서는 충분하지 않은가

파일이 이미 여러 branch, tag, merge commit에 섞여 있다면 특정 브랜치 하나만 되돌려 force-push해도 충분하지 않을 수 있다.

주의할 점은 아래와 같다.

- 다른 branch가 여전히 그 파일을 가리킬 수 있다
- tag가 오래된 commit을 계속 보존할 수 있다
- 다른 사람이 이미 clone 또는 fetch해 간 저장소에도 남아 있을 수 있다
- fork된 저장소가 있으면 거기에도 남아 있을 수 있다

즉, "제일 부모 브랜치까지 가서 다시 push"는 일부 상황에서는 통할 수 있어도 일반 해법은 아니다.

## History Rewrite 후 해야 할 일

히스토리를 재작성했다면 그 다음 단계도 중요하다.

### 1. 원격 저장소에 강제 반영

```bash
git push --force --all origin
git push --force --tags origin
```

remote가 여러 개라면 각각 다시 반영해야 한다.

### 2. 협업자 저장소 정리

히스토리가 바뀌었으므로 기존 clone을 그대로 쓰면 다시 오염될 수 있다. 가장 안전한 방법은 새로 clone하는 것이다.

### 3. 로컬의 오래된 참조 정리

로컬에서는 reflog나 dangling object 때문에 한동안 이전 객체가 남을 수 있다.

예를 들어 로컬 정리는 아래처럼 할 수 있다.

```bash
git reflog expire --expire=now --all
git gc --prune=now
```

이것은 로컬 관점의 정리일 뿐이고, 이미 유출된 비밀 자체를 무효화해 주지는 않는다.

## 서버에서도 바로 완전히 없어지는가

항상 즉시는 아니다.

이유는 아래와 같다.

- 원격 서버도 내부적으로 이전 객체를 일정 기간 보관할 수 있다
- 호스팅 서비스가 unreachable object를 즉시 제거하지 않을 수 있다
- 캐시, fork, 미러, 백업에 남아 있을 수 있다

즉, Git 논리상 참조가 끊겼더라도 운영 현실에서는 일정 기간 잔존 가능성을 고려해야 한다.

또한 `reflog`는 주로 로컬 개념이다. 다만 서버도 내부 보관 메커니즘을 가질 수 있으므로 "force-push 했으니 완전히 사라졌다"고 단정하면 안 된다.

## 민감정보가 들어갔다면 우선순위는 무엇인가

비밀값이 커밋된 경우에는 "Git에서 지우는 것"보다 "그 비밀을 더 이상 쓸 수 없게 만드는 것"이 먼저다.

우선순위는 아래처럼 잡는 것이 안전하다.

1. 키, 토큰, 비밀번호를 먼저 폐기 또는 교체한다
2. 그 다음 히스토리를 rewrite한다
3. 강제 push로 원격 히스토리를 교체한다
4. 협업자와 fork를 정리한다
5. 필요하면 호스팅 서비스나 보안 대응 절차를 밟는다

즉, 민감정보 사고 대응의 핵심은 삭제보다 회수 불가능성을 가정하고 무효화하는 데 있다.

## 언제 `git rm --cached`로 충분한가

아래 상황이라면 보통 `git rm --cached`와 `.gitignore`만으로 충분하다.

- 생성 파일을 더 이상 추적하고 싶지 않을 때
- 로컬 전용 설정 파일을 저장소에서 제외하고 싶을 때
- 민감하지 않은 파일이 실수로 관리 대상에 들어갔을 때

즉, 앞으로만 안 보이면 되는 문제라면 `--cached`가 적절하다.

## 언제 history rewrite가 필요한가

아래 상황이라면 history rewrite를 검토해야 한다.

- API key, 비밀번호, 인증서 같은 민감정보가 들어갔을 때
- 대용량 파일이 히스토리를 오염시켰을 때
- 법적 또는 정책상 과거 기록까지 제거해야 할 때
- 저장소 전체를 정리하며 특정 경로를 역사에서 제거해야 할 때

즉, "앞으로 안 보이면 되는가"가 아니라 "과거에도 남아 있으면 안 되는가"가 판단 기준이다.

## 짧은 정리

- `git rm --cached`는 추적 중단이다
- 과거 commit에서는 파일이 계속 보일 수 있다
- 완전 삭제는 history rewrite 문제다
- force-push 후에도 서버나 clone에 잠시 남을 수 있다
- 민감정보는 삭제보다 먼저 폐기와 교체가 우선이다
